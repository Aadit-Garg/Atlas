# Resolution Architecture

Dependency resolution in Atlas is Capability-driven, not Worker-driven.

## The Problem with Direct Imports
If `JournalWorker` directly imports `SQLiteWorker`, the architecture becomes tightly coupled. Migrating to `PostgresWorker` requires rewriting `JournalWorker`.

## The Atlas Solution
1. `JournalWorker` imports `capability.storage.sql` (a concept defined by a Model).
2. The Atlas Runtime checks the Registry for any installed Worker that exports `capability.storage.sql`.
3. `SQLiteWorker` exports this Capability.
4. Atlas creates a **Session** linking the two, negotiating required permissions.
5. `JournalWorker` now communicates with `SQLiteWorker` over the returned binding.

This guarantees extreme modularity. The `JournalWorker` never knows if it's talking to SQLite, Postgres, or an in-memory test mock generated by Solon.
